个人博客最重要的并不是拥有一套复杂后台，而是让写作、预览、发布和长期维护形成一条足够短的路径。

<Callout title="这套方案的边界">
  所有文章都来自仓库内可信的 MDX 文件。它不接收外部投稿，也不把未经检查的用户输入交给 MDX 编译器。
</Callout>

## 为什么选择 Git 驱动

文章与代码放在同一个仓库后，内容修改可以获得版本历史、代码评审和可回滚能力。发布流程也不需要单独维护数据库或管理后台。

- 文件名决定稳定的文章地址
- frontmatter 描述标题、摘要、日期和标签
- `draft: true` 的内容只在开发环境预览
- 合并代码后由构建流程生成静态页面

## 内容模型保持精简

第一版只保留真正影响展示与发布的字段。阅读时长、标题目录和相邻文章都由系统计算，避免作者维护重复数据。

| 字段          | 作用                   |
| ------------- | ---------------------- |
| `title`       | 页面标题与 SEO 标题    |
| `description` | 列表摘要与搜索描述     |
| `publishedAt` | 排序、RSS 和结构化数据 |
| `tags`        | 主题筛选与相关文章发现 |
| `draft`       | 控制是否进入生产构建   |

## Server Components 负责正文

文章文件只在服务端读取和编译，浏览器不会下载 frontmatter 解析器或 MDX 编译工具。客户端只需要承担主题切换等真正的交互。

```ts
const article = await getArticleBySlug(slug);

if (!article) {
  notFound();
}
```

这种拆分让正文保持接近静态 HTML，同时仍然可以在可信 MDX 中复用经过设计的 React 组件。

### 构建阶段校验

缺少标题、日期格式错误或标签为空时，构建应直接失败。内容错误越早暴露，线上页面就越可靠。

## 发布清单

- [x] 创建 kebab-case 命名的 `.mdx` 文件
- [x] 填写必要 frontmatter
- [x] 在本地检查标题层级和代码块
- [x] 将 `draft` 改为 `false`
- [x] 提交 Git 并等待生产构建

文章系统并不需要一次拥有所有功能。先让写作到发布的主路径稳定，再根据真实文章数量决定是否增加分页、全文索引或系列页。
